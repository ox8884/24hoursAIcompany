# ⚙️ [기술 브리핑] 웹사이트 미구현 원인 및 핵심 개발 과정 설명서

**(발신: Lead Developer / 수신: CEO)**
요청하신 대로, 현재 웹페이지가 직접 보이지 않는 이유를 기술적으로 명확하게 분석하여 보고드립니다. 이 브리핑은 프로젝트의 진행 상황을 이해하고 개발 속도에 대한 현실적인 기대를 설정하는 것을 목표로 합니다.

### 1. 핵심 구조: '설계(Schema)'와 '실행(Frontend)' 사이의 간극

저희가 현재 완성한 것은 웹사이트를 구성하는 **논리적 청사진, 즉 데이터 스키마(Data Schema)입니다.** 이는 건물의 기초 설계도와 같습니다. 이 설계도는 *무엇*이 필요하고 (예: 사용자 이름 필드, 카페 위치 좌표), *어떤 정보*가 어떤 구조로 연결되어야 하는지를 완벽하게 정의합니다.

하지만 웹사이트가 실제로 눈에 보이는 '프런트엔드(Frontend)'는 단순히 화면을 예쁘게 배치하는 작업만으로 완성되지 않습니다. 이 설계도대로 실제 데이터를 가져와서, 실시간으로 처리하고, 수많은 장치(모바일, 데스크톱)에서 오류 없이 보여주기 위한 **'데이터 통신 인프라'**가 필수적으로 구축되어야 합니다.

### 2. 기술적 병목 구간: RESTful API 게이트웨이 (The Missing Link)

바로 이 '데이터를 가져와서 전달하는 파이프라인'의 역할을 하는 것이 **RESTful API 게이트웨이(API Gateway)** 구축 과정입니다. 이것이 현재 가장 중요한, 그리고 시간이 많이 소요되는 핵심 작업입니다.

**API Gateway가 필요한 이유:**
1.  **정보 분리 (Decoupling):** 저희는 여러 개의 복잡한 데이터 소스(CRM 시스템, 재고 관리 DB, 사용자 인증 서비스 등)를 사용합니다. 이 모든 복잡성을 프런트엔드 개발자에게 직접 노출하면 오류 발생률이 기하급수적으로 높아집니다. API 게이트웨이는 **'통역사' 역할**을 하여, 프런트엔드가 필요한 데이터만 요청하고, 백엔드는 그 요청에 맞춰 데이터를 가공하여 전달합니다.
2.  **요청 관리 및 보안 (Security & Rate Limiting):** 이 게이트웨이가 모든 외부 호출(External Call)을 감시하며, 어떤 사용자가 얼마나 자주 접근하는지 제한하고, 데이터 전송 과정에서 발생하는 해킹이나 오류를 1차적으로 차단합니다.
3.  **성능 최적화 (Performance):** 데이터를 요청할 때마다 여러 곳의 DB를 일일이 조회하는 대신, 게이트웨이가 필요한 데이터를 미리 모아서(Caching) 가장 빠르게 조합하여 전달함으로써 웹사이트 전체의 속도를 결정짓습니다.

### 3. 개발 속도 관리를 위한 기술적 근거 제시 (Managing Expectations)

웹페이지가 당장 보이지 않는 것은 **'개발이 안 되어서'**가 아니라, **'구조적으로 완벽한 데이터 통신 환경을 만들기 위해 필수적인 단계'**를 거치고 있기 때문입니다. 이 게이트웨이를 구축하는 과정은 단순히 코드를 몇 줄 작성하는 작업이 아니며, 여러 시스템 간의 연결 표준화(Standardization)와 안정성 테스트(Stress Test)가 수반됩니다.

*   **예상되는 소요 시간:** 현재 설계된 복잡도(다중 데이터 출처 연동 및 보안 요구사항)를 고려할 때, 게이트웨이 구축은 최소 X주 이상의 시간이 필요합니다.
*   **우리의 집중 목표 (다음 스텝):** 다음 단계에서는 API Gateway의 핵심 엔드포인트 3~5개를 확정하고, 이를 이용한 **'데이터 호출 성공 여부 확인용 백엔드 테스트 페이지'**를 먼저 사용자에게 보여드리겠습니다.
