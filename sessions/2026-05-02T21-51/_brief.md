# 📋 작업 브리프

**원 명령:** sample.html 이라도 만들어서 내가 사이트 초안이라도 볼수있게 해주세요

## 요약
사용자가 즉각적인 시각적 피드백을 원하므로, 기술적인 이유를 다시 한번 설명하며 '와이어프레임/초안' 형태로 결과물을 제공해야 합니다. 단순 코드를 넘어선, 기대치를 관리하는 커뮤니케이션이 핵심입니다.

## 분배
- **📱 Secretary**: 사용자의 요청을 인지하고 즉시 응답할 초안을 작성한다. 기술적 복잡성(API Gateway) 때문에 완성된 페이지는 불가능함을 다시 한번 친절하게 설명하되, '구조화된 프로토타입'의 개념으로 전환하여 다음 주 목표인 Mock-up 시연이 곧 가능하다는 긍정적인 메시지를 전달하는 것이 핵심이다.
- **💻 Developer**: 실제 HTML 코드를 작성하되, 이 코드는 '뼈대(Scaffold)' 역할만 수행하도록 한다. 데이터가 들어갈 위치(예: 사용자 프로필, 메뉴 정보)에는 실제 데이터를 대신하는 플레이스홀더 주석(<!-- API Placeholder: ... -->)을 명확히 삽입하여, 이 초안이 아직 동적이지 않은 구조체임을 시각적으로 증명해야 한다.
- **🎨 Designer**: 개발자가 만든 뼈대 코드에 브랜드의 핵심 컬러와 타이포그래피를 적용하여 '초안(Draft)'처럼 보이게 디자인한다. 단순히 코드를 예쁘게 꾸미는 것이 아니라, 사용자의 시선을 사로잡을 중요한 요소(CTA 버튼, 헤드라인)에 디자인적 강조점을 주어 실제로 웹페이지가 있다는 착각을 불러일으키되, 기술적인 기대치를 낮추지 않는 수준으로 완성도를 높여야 한다.
